# Training Pipeline

> How synthetic data, structured vocabulary, and a small transformer come together.

## Overview

The training system converts the engine's deterministic logic into learnable examples for a transformer model. The model's job is to replicate the engine's reasoning — but faster, and generalizable to inputs the engine hasn't seen.

```
820 roots × 10 intents × 10 patterns × modifier variations
    ↓
  18,543 clean training examples (JSONL)
    ↓
  Transformer (1.5M – 19M params)
    ↓
  Learned model that maps serialized algebra tokens → actions
```

## Data Generation (v3)

The v3 data generator (`src/training/generate-v3.ts`) produces a clean, unified corpus of 18,543 examples — general algebra and agent tool-routing combined — with **zero UNK risk** (every token in the data exists in the vocabulary).

The generator:

1. Iterates over all 820 roots across 29 domains
2. Combines each root with intents and patterns
3. Runs the engine deterministically to get the correct action
4. Adds agent tool-routing examples for 3 domains (telecom, banking, healthcare — 64 tools)
5. Serializes input and output into the algebra vocabulary
6. Validates every token against `vocabulary-v3.json` before writing

Each example is a JSON line:

```json
{
  "input": "<BOS> I:send R:قرر RL:q-r-r P:patient MK:time MV:time:tomorrow <EOS>",
  "output": "<BOS> ACT:send R:قرر D:cognition CONF:high <EOS>"
}
```

Agent examples include TOOL and NEXT tokens:

```json
{
  "input": "<BOS> I:seek R:حسب RL:h-s-b P:patient <EOS>",
  "output": "<BOS> ACT:query TOOL:check_balance D:telecom CONF:high NEXT:report <EOS>"
}
```

The engine is the **teacher**. It produces the ground truth labels. The model is the **student** learning to produce the same outputs.

## Vocabulary Design

The base vocabulary defines 1,756 structured tokens. The v3 effective vocabulary is **1,871 tokens** — a tight, clean set with no open-ended literal tokens. See [vocabulary.md](vocabulary.md) for the base token design. Key design decisions:

### Dual root representation

Every root appears twice: as Arabic script (`R:قرر`) and Latin transliteration (`RL:q-r-r`). This redundancy gives the model two signals to identify the root — useful when one form has been corrupted or when the model is uncertain.

### Modifier key-value separation

Modifiers use two tokens each: a **key** token (`MK:time`) followed by a **value** token (`MV:time:tomorrow`). This lets the model learn that modifier keys and values are different kinds of information, and that the key predicts the type of the next token.

### Confidence in the output

Output sequences include `CONF:high`, `CONF:medium`, or `CONF:low`. This teaches the model to reproduce the engine's confidence — high when a rule matched, low when it fell back to the default "process" action.

## Serialization

The serializer (`src/training/serializer.ts`) converts between engine structures and token sequences:

### `serializeInput(token: AlgebraToken)`

```
AlgebraToken {
  intent: "send",
  root: "قرر",
  rootLatin: "q-r-r",
  pattern: "patient",
  modifiers: [{ key: "time", value: "tomorrow" }]
}
    ↓
"<BOS> I:send R:قرر RL:q-r-r P:patient MK:time MV:time:tomorrow <EOS>"
```

### `serializeOutput(result: ReasoningResult, domain: string)`

```
ReasoningResult {
  actionType: "send",
  confidence: 0.9,
  token: { root: "قرر", ... }
}
    ↓
"<BOS> ACT:send R:قرر D:communication CONF:high <EOS>"
```

### `deserializeToToken(tokens: string[])`

Reads model output tokens back into an `AlgebraToken` for the engine to process.

## Model Architecture

We trained three model sizes on the same v3 data to test whether capacity matters:

| Parameter        | Small     | Medium    | Large     |
| ---------------- | --------- | --------- | --------- |
| d_model          | 128       | 192       | 384       |
| Attention heads  | 4         | 4         | 8         |
| Encoder layers   | 3         | 4         | 6         |
| Decoder layers   | 3         | 4         | 6         |
| Feed-forward dim | 256       | 384       | 768       |
| Vocabulary       | 1,871     | 1,871     | 1,871     |
| Total parameters | **1.47M** | **3.69M** | **19.2M** |

### Why So Small?

GPT-4 has ~1.7 trillion parameters. LLaMA-7B has 7 billion. Our **small model has 1.47 million** — **4,760× smaller than the smallest LLaMA** — and matches the large model's accuracy.

This works because:

1. **Small vocabulary** — The embedding table is 1,871 × 128 = 239K params (small model). In LLaMA, the embedding table alone is 12.3M params.

2. **Structured tokens** — Every input token carries meaning. The model doesn't waste capacity learning that "the" is an article or that "report" is a noun. Those facts are pre-encoded.

3. **Deterministic teacher** — The training data is generated by a correct engine. There are no labeling errors, no ambiguous examples, no contradictions. The model sees clean signal.

4. **Bounded output space** — The model only needs to predict from 1,871 possible tokens, not 100K. The softmax over the output vocabulary is cheaper and more precise.

### What the Model Learns

The model does NOT learn:

- What Arabic roots mean (that's in the root database)
- What actions to take (that's in the engine's 74 rules)
- How to parse modifiers (that's in the encoder)

The model DOES learn:

- How to map noisy/novel inputs to the right algebraic tokens
- How to generalize from seen root/intent/pattern combinations to unseen ones
- How to interpolate between similar roots in embedding space
- When to be confident vs uncertain

The model learns to **speak the engine's language** — not to reason from scratch.

## Training Results (v3)

All three models trained on the same v3 dataset with the same random seed (42).

| Metric                | Small          | Medium         | Large          |
| --------------------- | -------------- | -------------- | -------------- |
| Training examples     | 18,543         | 18,543         | 18,543         |
| Effective vocabulary  | 1,871 tokens   | 1,871 tokens   | 1,871 tokens   |
| Train / Val split     | 15,762 / 2,781 | 15,762 / 2,781 | 15,762 / 2,781 |
| Batch size            | 256            | 256            | 256            |
| Epochs (early stop)   | early          | 34             | 29             |
| Best val loss         | —              | 0.3439         | —              |
| **Best val accuracy** | **89.7%**      | **89.8%**      | **89.5%**      |
| **TOOL accuracy**     | **41.5%**      | **42.7%**      | **40.1%**      |
| **NEXT accuracy**     | **100%**       | **100%**       | **100%**       |
| Model file size       | 5.8 MB         | 14.2 MB        | ~75 MB         |
| Training time/epoch   | ~2.5s          | ~7.7s          | ~13s           |
| Training hardware     | Colab T4       | Colab T4       | Colab T4       |

### The Key Finding

**All three models converge to the same ~89.5–89.8% accuracy ceiling.** A 1.47M parameter model matches a 19.2M parameter model — 13× fewer params, 13× smaller file, 5× faster training. The structured vocabulary is doing the reasoning, not the model capacity.

This is the thesis proof: **algebra-derived tokens transfer the semantic structure from the vocabulary into the model for free**, so additional parameters have nothing useful to learn.

### Which Model to Train

**Ship the small model (1.5M params, 5.8 MB).** It matches the large model's accuracy at a fraction of the size. The medium (3.7M) is available as a sanity check — if it scores higher than the small on your data, the small model is under-trained, not under-sized. The large (19M) training script has been removed; it was only needed to prove that scaling doesn't help.

### Convergence Profile (Medium model)

```
Epoch  1: 34.8% → 51.1%  (cold start)
Epoch  2: 69.5% → 82.1%  (rapid learning)
Epoch  5: 88.1% → 88.1%  (approaching plateau)
Epoch 13: 90.4% → 89.8%  (plateau begins)
Epoch 24: 90.8% → 89.8%  (best checkpoint)
Epoch 34: 91.0% → 89.7%  (early stop, patience=10)
```

The model reaches 82% by epoch 2, then grinds to ~89.8% by epoch 13, and never escapes. The gap between train (91%) and val (89.8%) is small — this is not overfitting. The remaining ~10% error is genuine ambiguity in the data.

### Agent Token Accuracy

Unlike the old training data (which scored 0% on TOOL tokens), the v3 clean data produces meaningful tool-routing learning:

- **TOOL: ~41–43%** — The model is learning which tool to invoke, though domain ambiguity limits accuracy. Many roots (e.g., سأل "ask") map to tools across multiple domains.
- **NEXT: 100%** — Next-step prediction (confirm, report, escalate) is fully learned across all model sizes.
- **STEP: 0 tokens in eval** — The v3 data doesn't emit STEP tokens; this is a data gap, not a model failure.

### The ~10% Error Ceiling

The remaining errors are **domain prediction failures** — roots like سأل (ask), طلب (seek), and عمل (work) legitimately belong to multiple domains. The model must guess which domain the training label chose, and sometimes guesses wrong. This is an algebra design question, not a capacity question.
